Muito bem.

Estou atualmente no meu diretório Story, que também é

o diretório de trabalho para este projeto em particular.

E aqui vou novamente dar uma olhada no "git log" para ver quais commits anteriores nós temos.

E podem ver que temos dois commits anteriores e eu gostaria de os colocar no GitHub.

Para o fazer, a primeira coisa a fazer é criar um controlo remoto.

Portanto, está a dizer ao meu Repositório Git Local que criei um Repositório Remoto

algures na Internet e que quero transferir todos os meus commits para lá.

Portanto, o comando que vamos usar é "git remote add origin".

Agora, origin é simplesmente o nome do seu controlo remoto e, teoricamente, pode chamar-lhe o que quiser.

Pode chamar-lhe bacon se quiser, mas por convenção e é

altamente recomendável que mantenha simplesmente o nome de origem.

E a razão é porque a maioria dos programadores está habituada à nomenclatura convencional, o que

significa que será muito mais fácil para eles compreenderem o que se passa no seu

projeto quando o virem, em vez de você fazer algo completamente diferente, o que será

muito surpreendente e será muito mais difícil para as pessoas compreenderem o que se passa.

Portanto, "git remote add origin", e depois vamos colar o URL do nosso

repositório remoto no GitHub, e depois vamos em frente e carregamos em Enter.

Agora que o repositório remoto está criado, podemos enviar o nosso

repositório local para o nosso repositório remoto, que se chama origem.

Por isso, vamos dizer "git push -u origin main".

E o que esta linha de código faz é empurrar o seu repositório local para o repositório remoto usando

a flag u ou a opção u, que basicamente liga o seu repositório remoto e o seu repositório local.

E depois vamos empurrá-lo para o ramo remoto que se chama

origem, e vamos empurrá-lo para o ramo que se chama principal.

O ramo principal é simplesmente o ramo padrão ou o ramo principal de todos os seus commits.

E mais tarde, quando falarmos de ramificação, vamos falar mais sobre

o que é um principal e o que é uma ramificação.

Mas, por agora, estamos apenas a dizer que vamos fazer push do nosso repositório

local para este origin remote e vamos fazer push para o ramo principal.

Então, vamos em frente e pressionar Enter. E vai haver um pouco de

trabalho a ser feito em segundo plano, porque vai ter de carregar o

seu repositório local para o repositório remoto nos servidores do GitHub e, dependendo

do tamanho do seu repositório local, isto pode demorar vários períodos de tempo.

Agora, assim que virmos o nosso prompt, o cursor a piscar ou o sinal de dólar, isso

significa que o nosso push foi bem sucedido e, como podem ver, diz "branch main set

up to track remote branch main from origin". Parece muito confuso, mas se voltarem ao GitHub

e estiverem na mesma página, tudo o que têm de fazer é carregar no comando R

para atualizar ou simplesmente premir o botão de atualização e podem ver que todos os nossos

ficheiros estão agora alojados no GitHub e estão completos com todas as nossas mensagens de confirmação.

ramo principal, que atualmente só tem dois pontos de salvaguarda ou dois commits.

Depois, quando estivermos prontos, podemos fazer o commit dos nossos ficheiros ou do nosso save point para o nosso repositório local.

Assim, depois de ter feito alguns commits, tem esta linha de tempo

de vários commits e a isto chama-se o seu Ramo Principal.

Portanto, o ramo principal é o seu ramo principal de commits ou pontos

de salvamento, e é seqüencial, e é geralmente onde o seu progresso principal

é salvo ou confirmado. Mais tarde, introduzimos a ideia do Repositório Remoto.

É importante notar que pode ter um repositório local completamente em paralelo com

um repositório remoto para verificar as diferenças entre eles, mas também pode sincronizá-los

ou enviar coisas do seu repositório local para o seu repositório remoto.

Assim, no nosso caso, o repositório local é o ficheiro git que temos dentro do

nosso diretório Story, e o repositório remoto é o GitHub, que aloja o nosso

código e também aloja todas as alterações que foram feitas entre os diferentes commits.

Então, temos este ramo principal de vários commits; primeiro commit, segundo commit, terceiro

commit no nosso repositório git local, e quando executámos o comando "git push",

então isso efetivamente empurrou todos esses commits, todas essas várias versões e

alterações, e pedaços de código para o nosso repositório remoto no GitHub.

Então é isso que o "git push" faz.
